# 📋 YYC³ Admin System 升级与优化方案

## 规划总则

> **执行有规划，规划有节点，节点有目标，目标可评估**

基于 [00-项目现状审核报告](./00-项目现状审核报告.md) 的审计发现，制定以下分阶段升级与优化方案。

---

## 一、总体目标

将 YYC³ Admin System 从当前「功能完整但存在安全隐患和依赖管理问题」的状态，升级为「五高架构（高可用、高性能、高安全、高扩展、高智能）」的生产就绪系统。

**核心指标**:
- 安全评分: 55 → 90+
- 依赖管理: 40+ `latest` → 0 `latest`（全部锁定版本）
- 构建质量: 跳过 TS/ESLint 检查 → 全量检查通过
- 包管理器: npm → pnpm（统一工作空间）

---

## 二、阶段划分

| 阶段 | 名称 | 目标 | 交付物 | 状态 |
| --- | --- | --- | --- | --- |
| Phase 1 | 🔴 安全修复 | 消除所有 P0 安全风险 | 安全认证系统、JWT Token、CORS 修复 | ⬜ |
| Phase 2 | 🟡 依赖治理 | 锁定版本、统一 pnpm、清理无效依赖 | 干净的 package.json、pnpm-workspace | ⬜ |
| Phase 3 | 🟡 架构修复 | 解决配置冲突、启用构建检查 | 单一 next.config、TS/ESLint 全通过 | ⬜ |
| Phase 4 | 🟢 质量提升 | 测试覆盖、环境验证、代码规范 | 完善的测试套件、标准化配置 | ⬜ |
| Phase 5 | 🔵 子项目升级 | F-ktv 和 mobile-app 版本升级 | 子项目与主项目技术栈对齐 | ⬜ |

## 三、详细任务分解

### Phase 1: 安全修复（P0 - 紧急）

#### Task 1.1: 替换 Mock 认证为数据库认证

**目标**: 将硬编码 Mock 用户替换为 MySQL 数据库查询
**涉及文件**:
- `app/api/auth/login/route.ts`
- `lib/db/mysql.ts`
- `lib/security/encryption.ts`

**实施方案**:
1. 创建 `users` 数据库表结构（含 bcrypt 密码哈希）
2. 使用 `bcryptjs`（或 Edge Runtime 兼容的 `bcrypt`）替代明文密码比对
3. 修改 login route 从数据库查询用户
4. 添加登录失败次数限制和账户锁定机制

**验收标准**:
- [ ] 无硬编码用户数据
- [ ] 密码使用 bcrypt 哈希验证
- [ ] 登录失败有错误响应
- [ ] 数据库连接失败有降级处理

**依赖关系**: 无
**风险点**: Edge Runtime 不支持所有 Node.js crypto API，需确认 bcrypt 兼容性
**完成标记**: ⬜

---

#### Task 1.2: 实现真实 JWT Token

**目标**: 使用 `jose` 库生成和验证 JWT Token
**涉及文件**:
- `app/api/auth/login/route.ts`
- `lib/api/client.ts`
- `lib/security/` 新增 `jwt.ts`

**实施方案**:
1. 安装 `jose` 库（Edge Runtime 兼容的 JWT 库）
2. 创建 `lib/security/jwt.ts` 封装 Token 生成/验证
3. 使用 `JWT_SECRET` 环境变量签名
4. Token payload 包含: userId, role, permissions, exp
5. API 中间件验证 Token 有效性

**验收标准**:
- [ ] Token 使用 RS256 或 HS256 签名
- [ ] Token 包含过期时间
- [ ] API 请求验证 Token 有效性
- [ ] Token 过期后正确返回 401

**依赖关系**: Task 1.1
**风险点**: 需确保 JWT_SECRET 在生产环境已配置
**完成标记**: ⬜

### Phase 2: 依赖治理（P1 - 重要）

#### Task 2.1: 锁定所有依赖版本

**目标**: 将所有 `latest` 标签替换为具体版本号
**涉及文件**:
- `package.json`

**实施方案**:
1. 先用 `pnpm install` 安装当前 `latest` 解析到的实际版本
2. 从 `pnpm-lock.yaml` 读取实际安装的版本
3. 将 `package.json` 中的 `latest` 替换为精确版本
4. 分类整理: dependencies vs devDependencies

**版本锁定原则**:
- 核心框架: 精确版本 (如 `"15.5.4"`)
- UI 库: caret 范围 (如 `"^1.2.3"`)
- 工具库: caret 范围

**验收标准**:
- [ ] `package.json` 中无 `latest` 标签
- [ ] `pnpm-lock.yaml` 已生成
- [ ] `pnpm install` 可复现安装
- [ ] `pnpm build` 成功

**依赖关系**: 无
**完成标记**: ⬜

#### Task 3.4: 实现真正的环境变量验证

**目标**: 替换空壳 `validateEnv` 为实际验证逻辑
**涉及文件**:
- `config/env.validator.ts`

**实施方案**:
```typescript
export function validateEnv(requiredVars: string[]): {
  isValid: boolean;
  missingVars: string[];
} {
  const missingVars = requiredVars.filter(
    varName => !process.env[varName]
  )
  if (missingVars.length > 0) {
    if (process.env.NODE_ENV === 'production') {
      throw new Error(`Missing required env vars: ${missingVars.join(', ')}`)
    }
    console.warn(`[ENV WARN] Missing: ${missingVars.join(', ')}`)
  }
  return { isValid: missingVars.length === 0, missingVars }
}
```

**验收标准**:
- [ ] 生产环境缺少必要变量时抛出错误
- [ ] 开发环境缺少变量时发出警告
- [ ] 所有验证函数有实际逻辑

**依赖关系**: 无
**完成标记**: ⬜
